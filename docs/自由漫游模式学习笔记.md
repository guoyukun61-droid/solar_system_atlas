# Solaris Atlas 自由漫游模式学习笔记

## 1. 这个功能解决了什么问题

普通行星详情模式的相机目标是确定的：选择某颗星体后，相机不断插值到预设位置并看向它。
自由漫游的目标不同，它需要把相机当作一艘持续前进的飞行器：

- 从冥王星轨道外侧进入，并朝向太阳系内部飞行。
- 根据鼠标或触摸输入连续改变朝向。
- 沿当前朝向自动前进。
- 接近天体时通过连续阻力减速，但不设置强制回推的空气墙。
- 与普通相机互斥，不能让两套逻辑同时修改相机。

项目因此增加了 `orbit` 和 `explore` 两种视图模式。

## 2. 从太阳系外围稳定出发

如果直接读取 `bodyPositions.current`，模式切换的第一帧可能仍然保存着普通视图的旧尺度坐标，相机因此会错误地出生在内太阳系。更稳定的做法是使用轨道公式直接计算海王星当帧方向，再把起点放到冥王星远日点之外：

```js
const neptuneAngle =
  neptune.initialAngle + elapsedTime * neptune.orbitSpeed;
const radialDirection = normalize(
  getOrbitPoint(neptune, neptuneAngle) * EXPLORE_DISTANCE_SCALE
);
const outerRadius =
  getOrbitOuterRadius(pluto) * EXPLORE_DISTANCE_SCALE + 12;
const launchPosition = radialDirection * outerRadius;
```

镜头初始朝向取径向向量的反方向，因此会从外太阳系朝太阳飞行。这里用的是展示单位，不是真实千米。

## 3. 鼠标位置如何变成转向

鼠标坐标先归一化到 `[-1, 1]`：

```text
x = 鼠标横向位置 / 画布宽度 × 2 - 1
y = -(鼠标纵向位置 / 画布高度 × 2 - 1)
```

屏幕中心附近设置死区，避免手部轻微移动让镜头一直漂移。越靠近屏幕边缘，转向速度越高。

水平方向使用世界竖直轴旋转，垂直方向使用相机自身的右向量旋转。项目使用四元数，而不是直接累加欧拉角，从而减少万向节锁和角度跳变问题。

## 4. 指数阻尼为什么适合动画

常见平滑公式是：

```js
const damp = 1 - Math.exp(-delta * strength);
current.lerp(target, damp);
```

`delta` 是当前帧耗时。相比固定写 `0.1`，这个公式在 30fps、60fps 或高刷新率屏幕上都能保持相近的动画速度。

项目将它用于：

- 起飞前相机位置与方向的平滑过渡。
- 鼠标输入的平滑。
- 视野角、近裁剪面和远裁剪面的变化。
- 速度线透明度。

## 5. 自动推进与调速

Three.js 相机默认朝本地 `-Z` 方向看。将这个方向应用相机四元数，就得到世界坐标中的前进方向：

```js
forward = (0, 0, -1).applyQuaternion(camera.quaternion);
```

每帧移动距离为：

```text
基础速度 × 推进倍率 × 阻力系数 × delta
```

滚轮或移动端按钮把推进倍率限制在 `0.3x–3.0x`。限制范围能避免速度接近零，也防止相机一步跨过较小行星。

## 6. 无空气墙的连续阻力

每颗天体只定义阻力影响范围，不再定义禁止进入的安全墙：

- 影响范围外阻力系数为 `1`。
- 越靠近天体，阻力按照接近程度的平方快速增加。
- 最低推进速度仍保留一个很小的正值，因此相机不会被锁死，也不会被瞬间弹开。

简化表达为：

```text
阻力 = 接近程度²
有效速度 = 基础速度 × max(1 - 0.965 × 阻力, 0.035)
```

这种方法类似进入强引力或高阻区域的感觉，但它不是空气动力学或引力模拟。
与直接重置相机位置相比，连续函数不会产生突然反弹和“撞上透明墙”的观感。

太阳的阻力影响范围比普通行星更大，因为太阳尺寸和辉光范围更大。

## 7. 漫游专用距离缩放

普通模式需要在一屏中展示太阳系结构，因此轨道距离经过较强压缩。
漫游模式将非太阳天体的位置向量统一乘以 `2.15`：

```js
explorePosition = orbitPosition * 2.15;
```

这种做法不会修改天体数据，也不会影响普通模式。关键是相机、天体、小行星带和目标识别必须使用同一套缩放坐标，否则视觉位置与碰撞/识别位置会错位。

## 8. 前方天体识别与二维投影

先计算相机前向量与“相机到天体”方向的点积：

```text
alignment = dot(cameraForward, targetDirection)
```

点积越接近 `1`，说明目标越靠近视野正前方。项目只识别前向锥体内的天体，再结合距离评分选择最佳目标。

选中目标后，通过 Three.js 的 `project(camera)` 将三维世界坐标转换成标准化屏幕坐标，再映射为 CSS 百分比：

```text
screenX = (ndcX × 0.5 + 0.5) × 100%
screenY = (-ndcY × 0.5 + 0.5) × 100%
```

HUD 的圆形锁定环、目标编号和距离信息因此能够跟随真实星球移动。目标靠近右边缘时，文字自动翻转到圆环左侧，避免移动端裁切。

## 9. 实例化小行星

漫游中使用 `InstancedMesh` 绘制低多边形岩块。多个小行星共享同一份几何体和材质，只提交不同的变换矩阵，相比为每块岩石创建独立 Mesh 更节省 draw call。

小行星分为两层：

1. 低亮度点状主小行星带，表现太阳系结构。
2. 稀疏的立体岩块航路层，分布在相机前方；飞到相机后方的岩块会重新投放到远处。

小行星不参与阻力计算，避免大量对象增加每帧碰撞检测成本。

## 10. 星空背景与视差

背景分成三类：

1. 跟随相机移动的银河天空球，保证转向时永远看不到背景边界。
2. 在较大空间范围内分布的多层星点，提供近远层次。
3. 跟随相机朝向的短线段，在漫游模式中模拟高速穿越感。

天空球只提供无限远背景。星点层不会每帧完全跟随相机，因此相机平移时能产生轻微视差；当相机飞得太远时，星点组再分段重定位，避免最终飞出星域。

## 11. 行星卫星与实例化渲染

月球、火卫一、火卫二、伽利略卫星、土卫六、海卫一等代表性卫星使用同一个 `InstancedMesh` 绘制。每颗卫星只保存母星、半径、轨道半径、速度、相位和颜色，每帧读取母星实时位置并叠加局部圆周运动。

这种方式减少了独立 Mesh 带来的 draw call，同时保留了足够清晰的空间层级。它仍是视觉化展示，并未模拟卫星轨道摄动或真实比例。

## 12. 本轮修改经验总结

- 多套相机控制必须互斥。否则普通聚焦相机和漫游相机会在每一帧互相覆盖，表现为抖动或闪烁。
- 模式切换第一帧不要依赖可能尚未更新的共享坐标；稳定出生点应从轨道公式直接计算。
- 如果产品要求无空气墙，就不能通过重置相机坐标来阻止接近；应使用连续阻力，并接受用户最终可以穿过模型。
- 漫游模式拉大天体距离时，渲染坐标、相机起点、目标识别和小行星带必须使用同一个缩放系数。
- 点状小行星带不能过亮或过大，否则会像整齐的像素横线；近景岩块应更少、更暗、分布更散。
- 前方目标不能简单选距离最近者，应同时考虑镜头朝向、可见范围和距离。
- 天空图片如果自带静态行星，不能直接作为背景，否则会与真实 3D 行星重复。
- 移动端转向应该基于手指拖动位移，而不是手指在屏幕上的绝对位置。
- 持续运行的 `useFrame` 中应复用 `Vector3`、`Quaternion` 等对象，避免每帧创建大量临时对象导致垃圾回收卡顿。
- HUD 必须与主浏览界面互斥。漫游时保留详情栏和选择条会遮挡视野，也会让交互意图不清晰。
- 速度仪表只保留瘦体数字时，仍应保留带无障碍标签的加减按钮，视觉简洁和可操作性可以同时成立。
- 卫星适合共享几何体和材质，通过实例矩阵表达各自的局部轨道，避免为小物体制造过多 draw call。

## 13. 可以继续学习的方向

- 用空间分区结构降低大量天体碰撞检测的复杂度。
- 使用曲线或航点系统制作自动巡航模式。
- 使用真实天文历表计算特定日期的行星位置。
- 使用 WebGPU 或自定义 Shader 构建体积星云和更真实的高速星线。
- 用有限状态机管理 `overview`、`focus`、`closeFocus`、`explore` 等复杂交互状态。
